The money math (reference)

  • Record payment: insert payments row (completed); amount_paid += pay; balance = max(0, total − amount_paid); status = balance ≤ 0 ? paid : partial; fires Joby invoice_paid when fully paid.
  • Send invoice: mints a payment_links token (90-day), attaches {baseUrl}/pay/{token} to the invoice, pulls job before/after photos, renders a PDF, then SMS + email.
  • Receipt: SMS/email "$amount_paid received for invoice #…" + optional default review_platforms link.
  • Follow-up: stop sets followup_stopped; resume clears it and resets the count.

⚠ Screen-level drift flags

  • ScanPay charge is a 404 — POST /payments/scanpay/charge is never registered (only scanpay-qr / -link / -status / -webhook exist). Web "Charge via ScanPay" always fails; Android targets the same missing path.
  • Web invoice signature is broken — web sends signature_data; the handler reads signature → 400 "Signature required". (Same wrong-key class as the Job-Detail signature.)
  • Inline vs dedicated: web does send / pay / sign / receipt in modals; Android routes each to a separate screen (InvoiceSend / Payment / InvoiceSign / Receipt).
  • Charge-payment visibility differs: web at any unpaid status; Android only at signed.
  • Add Item is Android-only on the detail screen; web edits items in InvoiceForm.
  • UNVERIFIED: /send explicit status='sent' write; whether /pay/:token resolves (registered link route is /api/payments/link/:token); Android addLineItems repo body.
