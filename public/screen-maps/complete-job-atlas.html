The money math (reference)

  • Guard: if source% + tech% > 100 → blocked (HTTP 400). Runs only on /complete, not the legacy status path.
  • Partner split: net = gross − parts − CC; sender = net·sender%, receiver = net·receiver%; then CC absorbed per who-pays (sender / receiver / split).
  • Tech three-slice: netProfit = gross − material_cost (from line items, not the complete-body parts); source = netProfit·source%, tech = netProfit·tech% (owner/admin → 0), company = the rest.
  • Writes in order: job_completion_details (pending) → jobs (completed) → tech_earnings (delete+insert) → material_reimbursements (if any).

⚠ Screen-level drift flags

  • Permissions (Phase 2b, 2026-06-07). Completion is gated jobs:edit_self + ownership (a technician completes their own job via requireJobOwnership('edit_self')). Approve Earnings in the resulting flow = accounting_earnings:full (owner/admin).
  • Partner-split money gap (web): RESOLVED (Phase 1). Web now renders the parts/CC form for partner jobs and sends the same body as Android → web-completed partner jobs produce the same job_completion_details numbers. No more overstated earnings.
  • Web "Payment Collected" dropdown: RESOLVED (Phase 1), the dead payment_method select was removed; web no longer sends it. Who-collected now rides payment_collected_by on partner jobs.
  • No web confirm surface, GET /jobs/:id/completion and POST /completion/confirm exist and are used by Android; web calls neither, so a partner sender on web can't confirm a split. (Unchanged.)
  • Two completion paths, web side closed (Phase 1): web's Status modal "Completed" now opens the Complete modal instead of POST /status?completed, so web completion always runs /complete (guard + partner split + completion record), mirroring Android. The legacy /status?completed branch still exists but web no longer uses it. Caveat: that branch also runs source-notify + inventory deduction + membership advance that /complete does not, Android skips them too; porting them into /complete is a backend follow-up.
  • Earnings = real collected money (Commit A, 2026-06-01). Gross unified to one helper getJobGross = SUM(completed payments, net of refunds) for the job (replaced 3 billed sources: latest-invoice, line-items chain, legacy SUM). /complete now branches: fully paid (SUM(balance_due)=0) → completed + earnings (dated today); balance remaining → holding + actual_end + completion record saved, NO earnings.
  • Commit B BUILT (2026-06-01, backend 2269f85): reconcileJobEarnings(jobId) is the payment-event trigger, wired into POST /invoices/:id/payment, POST /payments/:id/refund, and the ScanPay webhook. A later payment clearing a work-done holding job's balance auto-completes it + fires earnings on getJobGross dated to the clear-day + recomputes the partner split (kept pending). Refunds claw back (earnings drop, reopened balance flips completed→holding). Deposit-holding (actual_end null) is never auto-completed. Money model complete.
  • tip_amount column exists but neither surface captures a tip → always 0.
  • /complete is not transactional: a post-hook failure can leave the job completed with no tech_earnings row.
  • $0-collected completion → completed with $0 earnings (intended). Holding has two meanings: estimate-deposit (actual_end null) vs work-done-unpaid (actual_end set).
