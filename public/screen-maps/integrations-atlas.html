⚠ Screen-level drift flags

  • Two token tables; the code uses the out-of-band one. schema.sql:499 defines a generic integration_tokens, but quickbooks.js reads/writes qbo_connections (realm_id, access_token, refresh_token, token_expires_at, environment, company_name) — a different table whose DDL is not in committed db/. integration_tokens appears unused by the QBO route (UNVERIFIED whether anything else uses it).
  • One-way push only. Sync creates QBO Customers/Invoices/Payments and back-fills local qbo_*_id; nothing is pulled from QBO. Edits made in QBO never flow back.
  • Items/products not synced. Invoice lines all use a hard-coded QBO ItemRef {value:'1', name:'Services'}; no item-sync endpoint. Payments use a hard-coded PaymentMethodRef {value:'1'} and aren't applied to a specific invoice (just CustomerRef + TotalAmt).
  • Android OAuth doesn't return to the app. The callback redirect_uri is hard-wired to FRONTEND_URL (the website), so connecting from Android opens an external browser and lands on the web page; the Android screen reflects "Connected" only on its next status load.
  • Batched 100/run. Each entity sync is LIMIT 100 over un-synced rows, so large datasets need repeated runs (UI doesn't indicate "more remaining"). Disconnect deletes the local qbo_connections row but does NOT revoke at Intuit; local qbo_*_id markers persist. PUT /settings (sandbox/production toggle) exists but has no UI — environment is fixed to 'production'.
