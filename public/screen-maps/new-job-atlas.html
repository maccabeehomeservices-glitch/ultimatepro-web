⚠ Screen-level drift flags

  • Schedule timezone: RESOLVED (TZ 1/3+2/3+3/3). Each job carries its address timezone (jobs.job_timezone, resolved from lat/lng via local tz-lookup). Backend is the single converter (Path B): both surfaces send { scheduled_local, lat, lng }, the backend resolves the zone + converts wall-clock→UTC (luxon) + stores job_timezone; both display the stored instant in effective_timezone with a zone label (web date-fns-tz, Android java.time). 2pm stays 2pm end-to-end and cross-platform. Backfill script populates pre-existing jobs.
  • Status auto-flip on edit (Phase 1 F2b): PUT /jobs/:id now promotes unscheduled→scheduled when a date is added on edit. Reverse (date-clear) not reachable through PUT (COALESCE) — deferred.
  • Customer required vs auto-create: web blocks save without a customer; Android silently creates one from the typed name.
  • "Save & Send" RESOLVED (Phase 1 F3): both surfaces now notify the assigned tech via /roster-techs/notify-tech; the premature customer "on the way" dispatch at creation was removed.
  • Web Status field is dead on save — collected but never sent. Status only changes from Job Detail.
  • notify_sms/email/push sent by both but ignored by POST /jobs (not columns).
  • Android-only carried fields: source label + ticket #, linked_job_id, skip_duplicate_check, richer 4-option duplicate sheet.
