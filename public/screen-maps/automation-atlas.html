⚠ Screen-level drift flags

  • No create/edit/delete UI on either surface. Both web and Android are list + active-toggle only. The backend POST/full-PUT/DELETE for joby_rules exist (ownerOrAdmin) but are unreachable from the app, rules are seeded server-side (db/setup.js). Android's "Configure rules on the web app" empty-state hint is inaccurate (web can't create them either).
  • Trigger-label maps are cosmetic and partly wrong. Actually-fired events: estimate_approved, job_assigned, job_en_route, job_started, job_completed, job_cancelled, invoice_sent, invoice_paid. Web's TRIGGER_LABELS uses estimate_signed (never fired, real event is estimate_approved) + invoice_overdue/booking_received/job_status_changed (not fired by audited routes); Android's align better but lack job_assigned/job_started. Labels don't affect firing (rules match the stored trigger_event string).
  • executeRule ignores type and dispatch_logic. Firing only sends notifications per the notify_* flags + templates. A type:'auto_dispatch' rule does NOT auto-dispatch through fireJobyRules; autoDispatch() is a separate engine function called from the jobs route.
  • Web's ACTION_LABELS don't match the type enum. Web maps create_invoice/notify_office/… but joby_rules.type is CHECK-constrained to auto_dispatch/alert/reminder/cancel_flow, only auto_dispatch overlaps, so most rules show their raw type string. (Android's ACTION_LABELS are a different non-matching set.)
  • Delayed rules depend on cron. delay_minutes>0 writes a joby_pending notification row processed later (utils/cron.js); immediate rules fire inline. Neither screen surfaces pending/queued state.
